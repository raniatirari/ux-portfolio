The Challenge

One ticket
at a time

Our event platform only allowed users to register one attendee at a time. Enterprise clients and organizations frequently needed to purchase multiple tickets for colleagues — creating repetitive data entry, operational friction, and a rising volume of support requests.


At the same time, the events team required extensive attendee information for sponsorship reporting, approval workflows, and targeted communications. The challenge wasn't just about convenience, it was about resolving a fundamental tension between user efficiency and business-critical data needs.

Existing single-attendee flow
Problem Statement

How might we enable users to efficiently register multiple attendees for an event while preserving the business-critical data collection, approval workflows, and technical limitations of the existing platform?

Core Tension
User Need
Register multiple attendees in a single session
Minimize repetitive form completion
Intuitive ticket assignment across a team
A usable experience across desktop and mobile
Business Need
Collect detailed attendee data for each participant
Preserve sponsor and event qualification fields
Maintain approval and waitlist workflows
Reduce support volume driven by registration friction
Technical Reality
Legacy architecture: one registration per attendee
Limited engineering resources and timeline
Incremental solution — no full platform rebuild
Limited access to external participants for usability testing
Research & Discovery

Understanding
user behavior

A core part of discovery involved reviewing Hotjar session recordings to see how users actually moved through the registration flow. The pattern was clear: users mentally separated "buying tickets" from "assigning attendees," but the platform treated them as a single task.

100%
Redundant Re-Entry
Users re-entered organization and contact information for every single attendee — the platform stored no context between registrations.
↑
Drop-off at Re-Entry
Session recordings showed consistent drop-off and backtracking concentrated at repetitive data entry steps — not at payment or confirmation.
Split
Mental Model Gap
Users consistently treated purchasing and attendee assignment as two distinct tasks — the interface collapsed them into one, creating confusion at every step.
Design Strategy

How each constraint shaped a decision

01
Capture shared information once

Purchaser details entered once, reused across attendees — removing the primary friction without removing required data.

02
Separate purchaser from attendee

Distinct IA for buyer vs. participant — matching how enterprise users already think about group purchasing.

03
Progressive disclosure

Attendee-specific fields surface only when needed — reducing perceived complexity without hiding required data.

04
Backend compatibility first

Every pattern designed around the one-registration-per-attendee constraint — no engineering assumptions built in.

05
Framework, not a feature

Built to scale across event types and edge cases — a reusable pattern, not a one-off solution.

The Pivot

Build new, or build the bridge?

The first instinct was to build a dedicated group registration flow from scratch. But budget, engineering bandwidth, and the legacy architecture made that untenable on this timeline.

The call: design within the existing system as a deliberate bridge — solving the immediate user need now, without locking in assumptions that would conflict with the platform rebuild already underway.

AI-Assisted Validation

Synthetic testing
as risk reduction

With no budget for external research or A/B testing, I built a high-fidelity prototype and ran AI-assisted synthetic usability testing — simulating real registration tasks across key user archetypes to surface friction before development. Not a replacement for real users. A way to de-risk before committing engineering time.

Design

A framework built
to scale

Before settling on a final pattern, I evaluated multiple workflow models — testing which approach best aligned with the platform's limitations, could be delivered within our engineering constraints, and reduced complexity for users.

Exploration 01
Add Multiple Tickets — Layout Options
Comparing two structural approaches to adding tickets, plus variations on edit/delete interactions and summary layouts — testing which pattern minimized friction without adding screens.
Wireframe exploration · Option A vs. Option B · interaction variants
Exploration 02
Event Series Registration — Full Flow
Mapping the complete path for events that are part of a series — individual and group registrants, payment, confirmation, and edge cases like removing an attendee or hitting a registration cap.
User flow · Individual & group registrant paths · edge cases
Exploration 03
Single-Event Registration — Full Flow
The parallel flow for standalone events outside a series — built on the same individual/group structure so the pattern holds across event types rather than forking into one-off logic.
User flow · Standalone events · individual & group registrant paths
